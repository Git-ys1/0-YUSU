# Development History

## Why This File Is Intentionally Long

这条下位机线不是一条“从一开始就设计得很干净”的正式固件线，而是被硬件现象、
串口诊断、文档断言、分支回退、物理改线和反复上板一点点逼出来的。仓库里那些看起来
很碎的 `docs/PLAN`、`docs/SOFTWARE`、`docs/VERIFY`，很多并不是“水文档”，而是当时为了
让下一刀可判断、可回退、可对照而留下来的阶段性断言。

如果未来工程师只看今天的 `firmware/openrf1_motion_controller/`，会误以为项目是：

- 先定义好目录
- 再写驱动
- 再调 PI
- 最后整理仓库

真实顺序恰好相反：先在混乱中打通最小闭环，再靠大量断言和回退把混乱压缩成今天的
正式工程与正式文档入口。

## Phase Notes

下面的阶段笔记故意保留失败、反复和用户纠正。对下位机项目来说，这些不是情绪化历史，而是判断后续机械臂工作该怎么开局的证据。

## Phase 0: 仓库起点是治理，不是正式固件

最开始的下位机不是 RF1，也不是一套成熟的 STM32 工程，而是 Tyler / UNO 旧线与项目
纪律的建立。

这一阶段做对了两件事：

1. 把旧代码、README、版本命名、Git 纪律先纳入同一个仓库与同一套口径。
2. 一开始就把 `C` / `J` 双线、checkpoint、分支、回退写进项目总纲，而不是等出事故再补。

这解释了为什么后面 RF1 与机械臂两条线虽然过程很乱，但总能回到“这轮到底是哪条线、
从哪条基线出发”的讨论方式。

**Evidence**

- session `[0002]`：项目总纲直接把版本纪律、checkpoint、回退、`C/J` 双线写死。
- commits `930c224`, `09b1e0d`, `3f760e7`：仓库最早不是算法提交，而是基线导入与构建入口。

## Phase 1: UNO / Tyler 时代先冻结了控制边界

从 3 月中旬到 4 月上旬，下位机的大部分工作仍围绕 UNO、AFMotor、蓝牙协议、风机控制、
USB 手动控制和 `W` 协议展开。今天看这些内容像“旧时代”，但它们留下了三个长期资产：

1. `W` 连续控制思路不是 RF1 才有，而是在 UNO 时代就已经开始定语义。
2. 上位机与下位机的边界先被说清楚了，后面迁到 RF1 时不必同时重写协议。
3. 项目早早学会了把“协议冻结”和“硬件升级”拆成两件事。

这一段如果省略，会错误理解为 RF1 是从零发明了所有东西。实际上 RF1 继承的是协议、
调试和分层思想，不是从白纸开始。

**Evidence**

- commits `56b7447` through `4b2d6dc`
- `docs/SOFTWARE/C-2.2.4*`, `C-2.2.5A`, `C-2.2.8A`

## Phase 2: OpenRF1 bringup 是真正的架构切换

2026-04-14 到 2026-04-16 上午，是项目从 UNO 过渡到 RF1 的第一道硬分水岭。

用户在 `[0221]` 到 `[0227]` 明确提出了几件当时最现实的需求：

- ST-Link 和 TTL 已接好，希望能直接烧录和串口实测。
- 不只要代码，还要仓库根目录可执行的命令、复位方式、串口调试工具。
- 若 `M,1,300` 发出后电机不动，就不能宣称 bringup 已通。

这一阶段的特点不是“开始调闭环”，而是先证明：

- `CSR_RF1_READY` 能出现
- 最小 `M/E/STOP` 协议能打通
- 单轮 raw 控制能让电机真的转起来

它的重要性在于：从这里开始，团队第一次直接掌握 OpenRF1 上电机驱动、编码器、串口、
定时器、方向语义这些底层真值，而不是继续借用 UNO 时代的直觉。

**Evidence**

- commits `2498606`, `7d355b1`, `70b8602`
- session `[0221]`-`[0227]`

## Phase 3: 真值表阶段是 RF1 自证能力的出生点

真正让 RF1 进入“自写运行层”状态的，不是第一次轮子动，而是 2026-04-16 下午用户强行把
问题从“先跑起来再说”收口成“先写真值表，再回填代码”。

session `[0228]` 是这条线非常关键的一次纠偏：

- 用户把首轮实测现象完整列出：
  - `CN1: +700=反转, -700=不转`
  - `CN2: +700=不转, -700=正转`
  - `CN3: +700=正转, -700=不转`
  - `CN4: +700=不转, -700=反转`
- 明确否定“继续参考 `_work_2` 的成功实例来修 `_work_3`”这条路。
- 要求先把失败支线冻结，再开 `C-3.0.5B` 做完全自写的四路真值重建。

这一步极其重要，因为它把团队从“复制旧成功片段”拉回了“建立自己板子的真值表”。

随后的 `[0233]` 是另一个关键节点。用户明确写下“搞了 8 个小时终于好了”，并要求：

- 先冻结当前四路双向可控结果
- 再只修 `CN3/CN4` 的正向语义
- 把当前可控状态作为下一轮 `C-3.0.5C` 前的 checkpoint

`docs/VERIFY/C-3.0.5_openrf1_truth_table.md` 正是这段历史的凝结：它不是“普通验证记录”，
而是整个 RF1 自写底层第一次真正拥有了“先记录物理现象，再调整软件表述”的纪律。

**Evidence**

- session `[0228]`-`[0233]`
- commits `3090997`, `b108d43`, `4dc2700`, `9a1009d`
- `docs/VERIFY/C-3.0.5_openrf1_truth_table.md`

## Phase 4: CN1/CN3 编码器危机逼出了真正的硬件证伪纪律

如果说 `C-3.0.5B` 是驱动真值的出生点，那么 `C-3.0.8` 到 `C-3.1.4B` 就是编码器主线的
生死劫。

这段历史最值得记住的，不是“后来修好了”，而是项目是怎么被逼着学会不乱下结论的。

### 4.1 最开始并没有马上改脚

session `[0249]` 与 `[0250]` 仍然带着一种“是不是 TIM2/TIM5、remap 或 JTAG 释放没配对”的
软件怀疑。这在当时并不离谱，因为：

- `CN1 -> TIM5 + PA0/PA1`
- `CN3 -> TIM2 full remap + PA15/PB3`
- `CN3` 又天然牵涉 `SWJ/JTAG` 释放

但 `[0252]` 是一次非常关键的用户纠正：

- 不该先改脚。
- 先把公开分支、冻结结论、板级映射、输入电气模式、输入上拉、编码器模式矩阵系统性复核一轮。
- 只有最后一轮系统排查仍失败，才谈换脚或退回软件解码。

这条纠正非常重要，因为它让项目没有在证据不足时就把硬件编码器路线推翻。

### 4.2 软编码器分支是止损，不是胜利

`C-3.1.3` 这条 tracking-port soft-encoder 线今天很容易被误解成“曾经的主线方案”。
真实情况不是这样。

它的价值是：

- 在原生 TIM 路线结论未清时，先用循迹口软编码器把问题定位、止损、收集更多现象。
- 保证项目不会因为 `CN1/CN3` 暂时没数就整条链路停摆。

但它从来不是最终答案。后面项目明确要求把它保留成证据分支，而不是正式主线。

### 4.3 真正的收敛来自交叉硬件证伪

2026-04-19 的 `[0283]` 到 `[0288]` 非常能说明 Git 历史看不到、但会话证据必须保留的东西：

- 新电机到货
- 测 `CN1`、`CN3` 有没有反馈
- 把原实验室电机、新电机、`CN2` 控制组来回交叉
- 电池从 `7.4V` 换到 `12V`
- 最后把可疑白线线簇换成彩虹线再试

最终 `C-3.1.4B` 写死：

- 原生 TIM 主线软件成立
- `CN1/CN3` 的“有时有数有时没数”主要跟着电机编码器链、线簇/线序、接触状态走
- 白线线簇高度可疑，彩虹线替换后两路恢复非零反馈

这就是为什么后来的知识库不能把“软编码器分支”写成项目成就，它真正成就的是：
逼出了硬件证伪纪律。

**Evidence**

- session `[0249]`, `[0252]`, `[0275]`-`[0288]`
- commits `346f6a8` through `725e161`
- `docs/VERIFY/C-3.1.4B_openrf1_timer_final_convergence.md`

## Phase 5: 问题从“有无反馈”升级成“整车语义与同步”

到 `C-3.1.4C` / `C-3.1.4D` 之后，核心问题已经不是编码器会不会计数，而是：

- raw H 桥正负到底表示什么
- `W,a,b,c,d` 的四轮语义是什么
- 当前 `[CN1,CN2]` vs `[CN3,CN4]` 到底是左右分组还是前后分组
- 为什么实车“能前进，但偏；能有速度，但不能左右转；能跑，但不同步”

### 5.1 通道语义纠偏

session `[0302]` / `[0303]` 是这条线的重要认知修正：

- `CN1 = 左后`
- `CN2 = 左前`
- `CN3 = 右后`
- `CN4 = 右前`

用户明确指出：

- 不同增益不会“制造左右转”
- 左右转 / 横移来自 `W` 的四轮符号组合
- 若把 `[CN1,CN2]` / `[CN3,CN4]` 误认成前后组，实际上调的是左右侧

这是 RF1 进入“车体语义层”后的一次非常关键的抽象纠偏。

### 5.2 同步诉求引发了整段最难看的试错

2026-04-20 到 2026-04-22 这几天，项目最反复、最情绪化、也最容易被后人误读的一段历史，
正是同步诉求和底层抽象冲突的阶段。

用户连续强调的核心诉求非常稳定：

- 不想四轮各自一套玄学 PID
- 不想靠不同最小驱动、分段门槛、`2000-raw` 差异去“解释”不同步
- 只想要四轮同步、全正向、同一个 raw 语义

session 里的关键节点包括：

- `[0339]` / `[0340]`：明确质疑 `CN1/CN2` 走 raw、`CN3/CN4` 走 `2000-raw` 的不一致模型。
- `[0341]`：先设 checkpoint，再允许继续试。
- `[0346]`：明确要求回退到 `f826b3d` 看旧平滑基线，并提出“物理反接线能不能换来全正 raw”的思路。
- `[0349]`-`[0353]`：用户亲自把 `CN1`，后来连 `CN3` 的线反过来，并要求代码统一为 `1`。
- `[0356]`：把 `39c481f` 指定为“绝对基线”。
- `[0358]`：强烈纠正对 `SET/RESET` 的错误解读，强调之前的问题本质是 `CN1/CN3` 的电机符号一直是 `-1`，导致永远要走 `2000-raw`，从而破坏同步。
- `[0365]`：又一次非常有价值的现实提醒：有一轮看起来“代码不对”，最后发现其实只是忘了重新编译烧录。

这段历史留下的最重要成果，不只是 `C-3.3.1D`，而是三个纪律：

1. sacred checkpoint 必须真的可回退，不能一边说它是保险，一边在上面继续乱改。
2. raw 真值、编码器符号、`W` 语义、物理改线必须分层描述，不能掺着讲。
3. “同步”是正确目标，但如果在层级尚未拆清前就直接追同步，会把排障重新卷回混乱。

- session `[0297]`-`[0365]`

## Phase 6: 仓库治理不是“外围工作”，而是可交接性的前提

到 2026-05-19，用户已经明显不满足于“本地能跑”。`main` 的 README、入口文档、快速上手、
首页结构、Git 忽略规则，都被认为是重点仓库必须补齐的交接资产。

这一阶段的重要变化是：项目开始把“仓库对外可读性”视为和“电机能转”同等级的交付。

session `[0367]` 到 `[0388]` 显示：

- README 被要求重写成总述和跳转页
- 快速上手被单独拉出
- 首页不该出现的内容被要求删掉
- 大量不应跟踪的本地噪声被要求纳入 ignore
- RF1 PID 参数改动也要一起发布，而不是留在本地

这正是后面 `C-3.6.0` 能推进成功的前奏：用户已经不再接受 `_local`、临时脚本、
只有自己知道的目录结构。

**Evidence**

- session `[0367]`-`[0388]`
- commits `e13732c` through `1bf1281`

## Phase 7: 正式工程收口与机械臂分线

2026-06-07 是当前下位机条目的第二道总分水岭。

session `[0397]` 很直接地指出了此前最核心的工程治理问题：

- 用 `work2` 烧录 `work3`
- 最新成熟代码放在 `_local/car_move_lihaotian`
- 注释编码混乱
- 目录名 `_local` 完全不像正式主线

之后的动作不是“继续调算法”，而是：

1. 把真正参与编译的成熟 RF1 工程收口成 `firmware/openrf1_motion_controller/`
2. 只做注释、编码、结构清理，不在这轮改收敛后的控制行为
3. 清掉机械臂相关的大量未跟踪噪声，避免下一阶段再进同样的入口混乱

紧接着，session `[0410]` 到 `[0412]` 标志着下位机目标从“只有底盘”变成“底盘 + 机械臂”双主线：

- 用户明确决定先开始机械臂 STM32 调试
- 要把 `docs/1.STM32控制板源代码` 里的官方例程识别出来并烧到板上
- 随后又在 `[0412]` 要求：把这份最新机械臂官方例程放到正式位置，并为后续自研控制留独立空目录

最终 `C-3.7.0` 的成果不是“机械臂已经跑通”，而是：

- `firmware/mechanical_arm_official_baseline/` 成为机械臂官方冻结基线
- `firmware/mechanical_arm_controller/` 成为后续自研入口
- RF1 底盘和机械臂从此不再共用一个模糊的“资料区”入口

这一步对未来机械臂开发尤其重要，因为它正是吸取了 RF1 早期 `_local/work2/work3` 混乱历史后的组织化反应。

## Lessons By Stage

| Stage | Lesson | Future Use |
|---|---|---|
| Phase 0 / 1 | 版本纪律和 checkpoint 不是后期补丁，应该从项目初期写进规则 | 机械臂从第一天就要有官方基线、自研入口和回退点 |
| Phase 2 | OpenRF1 bringup 的第一目标是最小可观察闭环，不是完整架构 | 新硬件先跑 ST-Link、串口、单通道 raw，再谈高级协议 |
| Phase 3 | 当前板真值必须当前板实测，不能继承旧工程“成功语义” | 机械臂官方例程只能作为对照，不能自动变成自研真值 |
| Phase 4 | 硬件证伪优先于软件架构回退 | 舵机、供电、线束、端口异常必须先做物理对照 |
| Phase 5 | 同步、方向、编码器和物理改线必须分层，否则会越修越乱 | 机械臂角度正方向、动作组方向和坐标方向也要分层定义 |
| Phase 6 | 文档、忽略规则、脚本和正式目录直接影响硬件可复现性 | 不要把工程治理当成“外围整理” |
| Phase 7 | 新硬件线要先冻结官方基线，再开自研目录 | 机械臂不能重演 `_local/work2/work3` 多入口混乱 |

## Cross-Phase Patterns That Future Work Must Inherit

1. 这条线不是靠一次“正确设计”成熟的，而是靠 staged assertions 成熟的。很多看起来杂乱的文档，实际是下一轮判断的锚点。
2. 用户多次通过实物动作纠正软件叙事：挡住轮子、换电池、换电机、换线簇、物理反接线，这些都不会写进 C 代码，但决定了什么结论算数。
3. 分支、checkpoint、回退点在嵌入式项目里不是流程装饰，而是硬件排障的一部分。`f826b3d`、`167c3f8`、`39c481f` 都是阶段性保险丝。
4. 软绕行可以止损，但不能替代最终真值。`C-3.1.3` 之于软编码器，正如未来机械臂实验线之于官方基线：能救场，但不能混成正式主线。
5. 机械臂后续开发最该继承的不是某个 PID 参数，而是这套证据纪律：先冻结官方基线，再开自研目录；先做最小 smoke test，再谈融合。
