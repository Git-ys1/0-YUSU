# 技术决策

## Decision: 默认走宏录制器，而不是图像识别
**日期**: 2026-05 到 2026-06
**状态**: active

### 背景

最初需求探索过图像识别、固定屏幕区域、厨房小游戏状态机，以及 MaaNTE 风格自动化。用户后来发现目标游戏前期动作足够固定，MouseRecorder 式回放比图像识别更解决问题。

### 决策

让 `app.py` 默认启动宏录制器。保留通用/厨房/图像识别代码作为次级工具，但主工具优先优化录制、回放、快捷键、循环和连点器。

### 后果

工具变得更可用。代价是未来图像识别自动化应该作为独立任务重新定义，不要混在宏录制器主路径里。

## Decision: 录制绝对首个鼠标坐标
**日期**: 2026-05
**状态**: active

### 背景

回放不准，因为它从当前鼠标位置开始执行录制轨迹，而不是回到录制时的起点。

### 决策

`MacroRecorder` 在开始时写入全屏绝对坐标的 `mouse_origin` 事件。`MacroPlayer` 在开始延迟后先移动到第一个记录的鼠标事件，再回放后续事件。

### 后果

宏不再依赖当前鼠标位置。未来任何回放改动都必须保留这个行为。

## Decision: 使用 Windows 原生全局快捷键
**日期**: 2026-05
**状态**: active

### 背景

普通键盘监听在其他窗口或游戏里不稳定。

### 决策

在 `autogame/win_hotkeys.py` 中使用 `RegisterHotKey`。默认用 Shift + 功能键，并允许用户在界面修改。

### 后果

快捷键可靠性更好，并能检测冲突。首次启动后不要随意覆盖 `profiles/macro_settings.json` 中的用户设置。

## Decision: 游戏兼容输入保持可选
**日期**: 2026-05
**状态**: active

### 背景

有些游戏忽略普通桌面输入注入，但普通输入对非游戏窗口和测试仍有价值。

### 决策

UI 提供“游戏兼容输入”开关。可用时使用 `PyDirectInput`，不可用时记录日志并退回标准输入。

### 后果

同一 app 能服务普通桌面测试和游戏流程。新的自动化代码要在 `finally` 里释放所有按下的键/鼠标按钮。

## Decision: 连点器支持多键，而不是开多个连点器
**日期**: 2026-06-04
**状态**: active

### 背景

用户问连点器是不是只能连续点一个键。更实用的模型是一个连点器 + 按键列表 + 模式。

### 决策

`KeyClicker` 解析 `F J K` 或 `F,J,K` 这样的按键列表，并支持 `sequence`（轮流）和 `simultaneous`（同时）模式。

### 后果

UI 保持简单，同时覆盖常见高频按键场景。
