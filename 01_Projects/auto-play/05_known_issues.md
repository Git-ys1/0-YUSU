# 已知坑点

## Issue: 直接双击 `app.py` 可能调用错误 Python
**状态**: 已通过启动脚本缓解
**来源**: 项目 README 和启动脚本设计

### 症状

应用打不开，或闪一下就消失。

### 根因

Windows 的 `.py` 文件关联可能指向旧 Python，缺少项目依赖。

### 解决方式

使用 `.bat` 启动器定位 Conda 环境 Python，或退回 `conda-run.bat`。保留 `start_debug.bat` 用于可见日志。

### Codex 规则

交付 Windows 用户侧 GUI 工具时，要提供启动器和调试启动器，不要把 `python app.py` 当主体验。

## Issue: 全屏/管理员游戏可能忽略普通输入
**状态**: 已缓解
**来源**: 用户测试和项目 README

### 症状

宏在脚本窗口里能工作，在游戏中完全没反应。

### 根因

独占全屏、DirectInput 或管理员权限不一致会拦截输入注入。

### 解决方式

使用 PyDirectInput 游戏兼容模式、Windows 原生全局快捷键、必要时管理员启动，并建议切到无边框窗口/窗口化全屏。

### Codex 规则

游戏自动化“没反应”时，先检查权限、全屏模式和输入后端，不要马上重写业务逻辑。

## Issue: 回放不恢复首个绝对坐标会漂移
**状态**: 已解决
**来源**: 用户确认和 `macro_engine.py`

### 症状

宏按相同轨迹运动，但从当前鼠标位置开始，导致点击错误。

### 根因

初版把录制内容当作轨迹回放，没有强制恢复录制时的绝对起点。

### 解决方式

开始录制时记录 `mouse_origin`，回放开始延迟后先移动到第一个记录的鼠标坐标。

### Codex 规则

屏幕坐标宏必须记录并恢复绝对起点，再执行看起来像“相对轨迹”的事件流。

## Issue: Tkinter 回调里做阻塞操作会像卡死
**状态**: active rule
**来源**: UI 卡死反馈和最终架构

### 症状

点击控制面板后应用卡住。

### 根因

录制、回放、监听或输入循环不应跑在 Tk 主线程。

### 解决方式

录制/回放/连点器循环用 worker thread，消息通过 queue 回到 UI。

### Codex 规则

Tk 回调只做校验、启动/停止 worker，然后立刻返回。

## Issue: 固定流程上图像识别会过度工程化
**状态**: 已通过产品转向解决
**来源**: 用户纠正和当前 README

### 症状

图像识别厨房 UI 变得难配置、难调试：区域、备菜区、模板、状态机都需要额外工具。

### 根因

真实游戏流程比最初假设更固定；还没验证宏回放是否足够，就先上了视觉自动化。

### 解决方式

默认工具转为宏录制器，图像识别代码保留为次级/实验路径。

### Codex 规则

做用户侧自动化前，先判断流程属于固定轨迹、低分支状态机，还是高分支视觉 agent。选能工作的最简单层级。
